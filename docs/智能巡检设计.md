# 智能巡检设计

本文说明平台智能巡检能力的完整设计：采集什么、怎么采、怎么分析、怎么闭环。

## 1. 设计原则

1. **诚实采集**：只自动采集程序可真实获取的指标；应用层无法程序化获取的项目（证书有效期、账号权限、审计日志等）走人工录入通道，系统绝不编造数据。Agent 离线时监控曲线诚实降级为"无数据"，不生成假曲线。
2. **不依赖浏览器**：自动巡检由平台后端常驻守护线程执行，运维人员关闭浏览器后巡检与自愈照常运行。
3. **采集与分析分离**：Agent 端只做轻量采集与命令执行；全部智能分析（异常检测、根因、趋势、报告）在平台端完成，Agent 保持最小依赖。

## 2. 三层巡检体系

```
第一层  实时监测（5 秒级）
        Agent psutil 采集 CPU / 内存 / 磁盘 / IO / 网络 / 进程
        平台 Web 5 秒自动刷新 + 后端趋势采样入库
                    │ 异常
                    ▼
第二层  自动巡检与自愈（10 秒级，后端常驻）
        后端守护线程每 10 秒轮询全部服务器
        本地模型异常检测 → 告警聚合去重 → 根因规则匹配
        自动模式下直接调用 Agent /repair 执行修复并记录日志
                    │ 汇总
                    ▼
第三层  深度巡检与报告（按需 / 定期）
        Agent /inspect 采集 17 项系统巡检指标 + 人工录入项合并
        7B 多模态大模型按系统生成深度分析报告
```

## 3. 巡检指标：17 项覆盖

### 3.1 Agent `/inspect` 自动采集（`agent/agent.py`）

| 类别 | 指标 | 采集方式 |
|---|---|---|
| 进程 | 进程总数、内存占用 Top 10 进程（PID/用户/内存%/线程数） | psutil `process_iter` |
| 线程 | 全系统线程总数 | psutil 逐进程累加 |
| 连接 | ESTABLISHED 连接数 | psutil `net_connections`，非 root 受限时自动降级 `ss` 命令 |
| 端口 | 全部监听端口清单 | 同上 |
| 中间件 | Redis / MySQL / PostgreSQL / Oracle / Nacos / RabbitMQ / Kafka / ZooKeeper / Nginx / Tomcat / MongoDB / Elasticsearch / etcd 等运行探测 | 知名端口 × 监听端口交集推断 |
| 定时任务 | crontab 条目数 + /etc/cron.d + systemd timers | crontab -l / systemctl list-timers |

### 3.2 人工录入补充（平台 `/api/inspection/manual`）

程序无法可靠采集的应用层巡检项，由运维在平台"系统巡检项录入"表单按系统填写，与自动采集数据合并进报告：

- 证书过期时间、账号权限合规、审计日志留存
- ISC 登录限制、业务登录主页可用性等业务侧检查项

自动采集 + 人工录入合计覆盖需求文档要求的 **17 项巡检内容**。

### 3.3 实时指标（`/status`，5 秒级）

CPU 使用率（含每核）、内存使用率与可用量、磁盘使用率、磁盘 IO、网络收发速率、负载、运行时长、关键服务存活状态。

## 4. 后端常驻自愈守护（10 秒巡检）

平台端 `app.py` 内置守护线程，与浏览器完全解耦：

1. 每 10 秒轮询所有已接入服务器的 `/status`；
2. 本地模型判定指标异常或检测到故障注入标记 → 立即调用 Agent `/repair auto`；
3. 修复过程实时写入日志流，前端"修复执行过程"卡片可回看；
4. 手动/自动模式一键切换（`POST /api/auto-repair/daemon`），修复历史全部落盘并可镜像至 MySQL。

安全护栏：修复命令白名单化（10 类 playbook action），自定义命令过危险命令黑名单（禁 rm 系统目录、shutdown/reboot、mkfs、dd 物理盘、fork 炸弹）。

## 5. 智能分析链路

巡检数据进入平台后依次经过（模型细节见[《AI模型架构》](AI模型架构.md)）：

| 环节 | 模块 | 做什么 |
|---|---|---|
| 异常检测 | `models/ml_models.py` | 阈值 + 统计双通道判定异常，输出严重级别 |
| 告警聚合 | `services/alert_aggregator.py` | 60 秒窗口去重，按（服务器, 故障类型）聚合，避免告警风暴 |
| 根因分析 | 同上 | 6 条根因规则按优先级匹配（故障注入 / 内存连锁 / 磁盘满 / CPU 高 / 内存压力 / 连接数异常） |
| 趋势预测 | `services/trend_predictor.py` | 对历史采样做线性回归外推，输出未来 24h 趋势曲线 + 风险评分 |
| 修复决策 | `models/repair_decision_model.py` | ML 决策树 + DL 网络混合选择修复策略并预测成功率 |
| SOP 推荐 | `/api/sop/recommend` | 根据根因自动匹配知识库 SOP，支持多方案对比（风险/恢复时间/副作用） |

## 6. 巡检排程

`/api/maintenance/schedule` 基于服务器健康评分与历史故障频率生成差异化巡检计划：健康度差、故障多的服务器巡检频次自动提高，巡检项按风险侧重（如磁盘趋势异常的机器加做日志异常扫描）。执行结果回流 KPI 看板（`/api/dashboard/effectiveness`：修复总数、成功率、平均 MTTR、故障分布），形成"巡检 → 发现 → 处置 → 评估 → 排程优化"闭环。

## 7. 按系统深度分析报告

报告生成（`services/report_generator.py`）以"系统"为单位组织：

1. 按 `server.system` 字段将服务器归组为业务系统；
2. 在线节点实时拉取 `/inspect` 巡检数据（离线节点不取，避免造假），合并该系统的人工录入巡检项；
3. 交给 7B 大模型逐系统生成结构化中文分析：运行状况、巡检项逐条结论、风险点、处置建议；
4. 报告归档可导出，支持图片附件（监控截图经 7B 多模态图文分析后写入报告）。
